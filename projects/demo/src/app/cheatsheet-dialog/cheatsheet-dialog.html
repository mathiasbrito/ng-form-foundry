A form is described by one root nodeGroup. Every field is a node with a kind, a name (its key), and an optional label. Edit the schema JSON in a demo to see any of this change live.

Leaf types (type)

'string' · 'number' · 'boolean' · 'enum' (with enum: [...]).

On every node

nameIdentity — must equal the key in children. Required.
labelDisplay name (falls back to name).
descriptionHelper text.
defaultInitial value when none is supplied.
orderSort weight among siblings.

Optional · required · nullable

presence: trueOptional — the key may be absent. Rendered with an "Add …" affordance; a list materializes with its first entry and de-materializes when the last is removed. Omit it (or set required) for an always-present field.
nullable: trueThe value may be an explicit null (distinct from an absent key).
readOnly: trueDisplayed but not editable (e.g. a const).
required: trueLeaf must hold a value; a required choice must pick a case (mandatory).

How presence & required combine

They answer different questions — presence: may the key be absent? (whether a control exists at all); required: must a present value be filled? (validation, leaves only). For a leaf:

presencerequiredBehavior
——Always present; value may be left empty.
—✓Always present; must be filled (empty ⇒ invalid).
✓—Optional — "Add" to include, remove to drop. Once added it must be filled*: a materialized empty key would be a meaningless null.
✓✓Same as above for a leaf (presence already implies "fill once added"). This is the pair the transformer emits for an advisoryRequired field.

* unless nullable: true, which lets an added field stay null. Lists and containers have no required: a list's "always present" is just the absence of presence — use minItems for "at least N", a choice's mandatory for "must pick a case".

Validation

stringminLength · maxLength · pattern · format: 'email'|'uri'|'url'
numberinteger · min · max · multipleOf · radix: 2|8|16 (hex/oct/bin)
listsminItems · maxItems
mapkeyPattern · minEntries · maxEntries
groupminPresent · maxPresent (how many optional children are enabled)

Choice & Map

choice.casesA record of case-name → fields (or a single node). caseLabels names them in the picker; mandatory forces a selection.
map.valueThe node every entry conforms to (a leaf, group, list, choice, or nested map). Keys are user-entered.

Appearance (presentation only)

flattenRender the group's fields inline, without a wrapping card.
collapsedStart a group/map/choice section collapsed.
noBorderDrop the section border.
grid: { rows, cols }Lay fields on a CSS grid.
minFieldWidthAuto-fit equal columns at least this wide (e.g. "12rem").
booleanFields'beginning' | 'end' | 'default' — gather checkboxes into a compact row.
minTextFieldWidth / minNumberFieldWidthPer-type minimum widths in the flex flow.

Layout keys cascade to nested groups, list items, map entries, and choice cases; a child can override the pair it sets.
